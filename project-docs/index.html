01Project Overview

Atlas Journey is a trip planning application built for the Design and Analysis of Algorithms (DAA) course. It demonstrates three fundamental algorithm paradigms through a real-world travel planning scenario set in the fictional Atlas City.

Core Concept: A user explores Atlas City's 10 districts, plans a budget-optimized itinerary via 0-1 Knapsack, finds the shortest route via Dijkstra, and discovers top-ranked attractions via Heap Sort.

Project Goals

  • Demonstrate correct implementation of 0-1 Knapsack, Dijkstra, and Heap Sort
  • Provide an interactive, visually polished trip planning experience
  • Work offline with no external API dependencies
  • Operate as both a desktop application (Java Swing) and web application (HTTP Server + SPA)

User Workflow

  1. Login — Enter the application via the login screen
  2. Dashboard — View trip overview, stats, and quick actions
  3. Smart Itinerary — Set a budget, run 0-1 Knapsack to optimize attraction selection
  4. Navigation — Pick start and end locations, run Dijkstra for shortest path
  5. Discover — View all attractions ranked by Heap Sort composite score

DAA Relevance

0-1 Knapsack

Dynamic Programming — Budget optimization with optimal substructure

Dijkstra

Greedy + Priority Queue — Shortest path on weighted road graph

Heap Sort

Comparison-based Sorting — Max-Heap construction and extraction

Data Structures

2D DP Tables · Adjacency Lists · Priority Queues · Binary Heaps

Key Architectural Decisions

  • Dual-mode: Both desktop (Swing) and web (HTTP Server + SPA) share identical algorithm implementations
  • No external APIs: Map is SVG-based with embedded Atlas City data — no Google Maps dependency
  • SQLite: Self-contained database with zero configuration — ideal for offline use
  • Client-side map rendering: The web app renders maps via SVG, the desktop app uses Java2D canvas
  • Singleton services: Desktop services load data once and cache; web services load fresh per request

07Algorithm Analysis

07-A0-1 Knapsack — Budget Optimization

KnapsackService.java O(n · W) Time · O(n · W) Space

Why it is used: The 0-1 Knapsack algorithm solves the budget allocation problem: given a fixed budget (knapsack capacity) and a set of attractions (items) each with a cost (weight) and experience value (profit), select the subset that maximizes total value without exceeding the budget.

Where it is used: Smart Itinerary / Budget Planner page. Users set a budget amount, click "Optimize Budget," and the algorithm returns the optimal set of attractions.

Files involved:

  • src/algorithms/KnapsackService.java (desktop)
  • webapp/src/KnapsackService.java (web)
  • src/model/KnapsackResult.java (result model)
  • src/ui/BudgetPlannerPanel.java (desktop UI)
  • webapp/frontend/app.js lines 1223-1343 (web UI)

Step-by-step execution:

  1. Fetch all 12 attractions from database via getAllAttractions()
  2. Build 2D DP table dp[n+1][W+1] initialized to 0
  3. For each item i (1 to n), for each budget w (0 to W):
    dp[i][w] = max(dp[i-1][w], dp[i-1][w-cost_i] + value_i) if item fits
  4. Backtrack from dp[n][W]: if dp[i][w] != dp[i-1][w], item i was selected
  5. Return KnapsackResult with selected items, total value, total cost

Pseudocode:

function solveKnapsack(items, budget):
    n = items.length
    W = budget
    dp = new int[n+1][W+1]

    // Build DP table
    for i = 1 to n:
        for w = 0 to W:
            dp[i][w] = dp[i-1][w]
            if items[i-1].cost <= w:
                dp[i][w] = max(dp[i][w], dp[i-1][w - items[i-1].cost] + items[i-1].value)

    // Backtrack to find selected items
    selected = []
    w = W
    for i = n down to 1:
        if dp[i][w] != dp[i-1][w]:
            selected.add(items[i-1])
            w -= items[i-1].cost

    return KnapsackResult(selected, dp[n][W], totalCost, budget)

07-BDijkstra — Shortest Path Routing

DijkstraService.java O((V + E) log V) Time · O(V) Space

Why it is used: Dijkstra's algorithm finds the shortest path between two locations on the Atlas City road network. It guarantees the minimum total distance when edge weights are non-negative.

Where it is used: Navigation / Route Planner page. Users select start and end locations from dropdowns, click "Find Shortest Path," and the shortest route is computed and highlighted on the map.

Files involved:

  • src/algorithms/DijkstraService.java (desktop)
  • webapp/src/DijkstraService.java (web)
  • src/model/PathResult.java (result model)
  • src/model/Edge.java (edge model)
  • src/ui/RoutePlannerPanel.java (desktop UI)
  • src/ui/components/MapPanel.java (route rendering)
  • webapp/frontend/app.js AtlasCityMap class (SVG route rendering)

Graph representation: Adjacency list: Map<String, List<Edge>>

  • Nodes: 20 (10 districts + 10 junctions)
  • Edges: 70 directed (35 undirected pairs)
  • Weights: Integer values (5-20) representing distance

Step-by-step execution:

  1. Load all edges from database; build adjacency list
  2. Initialize all distances to Integer.MAX_VALUE, set start distance to 0
  3. Add start node to min-priority queue ordered by distance
  4. While queue is not empty:
    a. Poll node u with smallest distance
    b. If u == end, break
    c. Skip if already visited (stale entry)
    d. For each neighbor v of u: relax edge if dist[u] + weight < dist[v]
  5. Reconstruct path by walking prev map from end to start
  6. Return PathResult with node sequence, total distance, success flag

Pseudocode:

function findShortestPath(graph, start, end):
    dist = Map: node -> INFINITY
    prev = Map: node -> null
    visited = Set
    pq = PriorityQueue(ordered by dist)

    dist[start] = 0
    pq.add(start)

    while pq is not empty:
        u = pq.poll()
        if not visited.add(u): continue
        if u == end: break

        for each edge (u -> v, weight) in graph[u]:
            if visited.contains(v): continue
            newDist = dist[u] + weight
            if newDist < dist[v]:
                dist[v] = newDist
                prev[v] = u
                pq.add(v)

    if dist[end] == INFINITY: return "No path"

    path = []
    curr = end
    while curr != null:
        path.prepend(curr)
        curr = prev[curr]

    return PathResult(path, dist[end])

07-CHeap Sort — Attraction Rankings

HeapSortService.java O(n log n) Time · O(1) Auxiliary Space

Why it is used: Heap Sort provides a guaranteed O(n log n) comparison-based sort with O(1) auxiliary space (in-place). It sorts attractions by a composite score to produce a ranking from best to worst.

Where it is used: Discover / Rankings page. All 12 attractions are sorted by composite score (rating × 10 + value) and displayed in ranked order.

Files involved:

  • src/algorithms/HeapSortService.java (desktop)
  • webapp/src/HeapSortService.java (web)
  • src/model/Attraction.java (getScore() method)
  • src/ui/RankingsPanel.java (desktop UI)
  • webapp/frontend/app.js discover page (web UI)

Score formula: rating × 10 + value

Tie-breaking: Alphabetical by name (via compare() method)

Step-by-step execution:

  1. Fetch all 12 attractions from database
  2. Build max-heap: heapify all non-leaf nodes from n/2 - 1 to 0
  3. For i = n-1 down to 1: swap root (largest) with arr[i], heapify reduced heap
  4. Result array is now sorted in ascending order
  5. Reverse to get descending order (highest score first)

Pseudocode:

function heapify(arr, n, i):
    largest = i
    left = 2*i + 1
    right = 2*i + 2

    if left < n and arr[left].score > arr[largest].score:
        largest = left
    if right < n and arr[right].score > arr[largest].score:
        largest = right
    if largest != i:
        swap(arr[i], arr[largest])
        heapify(arr, n, largest)

function heapSort(items):
    arr = items.toArray()
    n = arr.length

    // Build max-heap
    for i = n/2 - 1 down to 0:
        heapify(arr, n, i)

    // Extract elements
    for i = n-1 down to 1:
        swap(arr[0], arr[i])
        heapify(arr, i, 0)

    // Reverse for descending
    return reverse(arr)

08Map System

Atlas City is a fictional city with 10 districts and 10 road junctions. The map is entirely self-contained—no external mapping API is used.

Desktop Map (MapPanel.java)

  • Rendering: Java2D canvas with layered drawing
  • 20 nodes: Each has normalized (0-1) x,y coordinates mapped to panel dimensions
  • 28 road segments: Hardcoded pairs of connected nodes
  • Rendering layers: Background gradient → Grid → Water (lake, river, pond) → District overlays → Roads → Junctions → Route → Markers → Labels → Zoom indicator
  • Route: Three-layer: glow (14px semi-transparent blue), main (4px green), center (1.5px white)
  • Animation: White dots move along route path (time-based phase calculation)
  • Zoom: 0.5x to 2.0x with pan offsets

Web Map (AtlasCityMap class in app.js)

  • Rendering: SVG with 1000x800 viewBox, responsive scaling
  • 22 intersection nodes: With id, name, x/y coordinates
  • 45 edges: Index-based connections between intersections
  • Filters: SVG blur filters for route glow effects
  • Interactive: Scroll-wheel zoom, hover tooltips on district pins, click navigation
  • Route animation: requestAnimationFrame-driven dashed line movement
  • Backend mapping: Maps DB location names to SVG intersection indices for route rendering

How Dijkstra Integrates with the Map

  1. User selects start and end locations from dropdowns (populated from DB nodes)
  2. Desktop: DijkstraService.findShortestPath() returns a List<String> of node names
  3. Desktop: MapPanel.setRoute() receives the node list and draws the route overlay
  4. Web: AtlasCityMap.setRoute() maps backend names to SVG intersection indices, computes road-following path via client-side Dijkstra, renders SVG route overlay
  5. Route is drawn as a highlighted path through the road network—not a straight line

11Design Decisions

Why SQLite? Zero Configuration
  • Self-contained: Single file (trip.db) with no server process
  • Zero configuration: No connection strings, user accounts, or schema management tools needed
  • Portable: Database travels with the application—ideal for demos and academic projects
  • Embeddable: JDBC driver is a single JAR file (sqlite-jdbc-3.42.0.0.jar)
Why a Local Map? No API Dependency
  • No API keys: Google Maps, Mapbox, and OpenStreetMap require API keys and internet access
  • Offline-ready: The map works without any network connectivity
  • Controlled data: We define the fictional city (Atlas City) exactly as needed for the algorithms
  • Pedagogical clarity: Students can see the road graph structure directly
Why Java + HTML/CSS/JS? Dual-Mode Application
  • Desktop (Java Swing): Required for demonstrating Java OOP, MVC, and desktop application architecture
  • Web (HTML/CSS/JS): Demonstrates client-server architecture, REST APIs, and modern SPA development
  • Shared algorithms: Both versions use the same algorithm logic, demonstrating code reuse
  • Academic completeness: The dual-mode approach shows versatility
Why Dijkstra (not BFS)? Weighted Graph

The Atlas City road network has weighted edges (distances from 5 to 20). BFS finds the shortest path in terms of number of edges, not total weight. Dijkstra guarantees the minimum total distance on weighted graphs with non-negative weights.

Why Heap Sort (not Merge Sort)? In-Place Sorting

Heap Sort has O(1) auxiliary space vs Merge Sort's O(n). Both have O(n log n) time complexity in all cases. Heap Sort was chosen to demonstrate the max-heap data structure and its applications beyond priority queues.

Why 0-1 Knapsack (not Fractional)? Real-World Applicability

You cannot visit "half" of a museum. The 0-1 Knapsack (each item taken once or not at all) accurately models real trip planning decisions. It also demonstrates dynamic programming with optimal substructure and overlapping subproblems.

Algorithms (Core)

+Why Dijkstra instead of BFS for shortest path? Algorithms
BFS finds the shortest path in terms of number of edges (unweighted graphs). Atlas City roads have weighted edges (distances 5-20 km). Dijkstra's algorithm handles weighted edges by using a priority queue to always expand the closest unvisited node, guaranteeing the minimum total distance. BFS would find a path with the fewest road segments, which could be much longer in total distance.
+Why is Dijkstra considered a "greedy" algorithm? Algorithms
Dijkstra's algorithm makes a locally optimal choice at each step by selecting the unvisited node with the smallest known distance. This greedy choice works because edge weights are non-negative—once a node's distance is finalized (popped from the priority queue), no shorter path can be found later. This is the greedy choice property.
+What is the time complexity of Dijkstra with a binary heap? Algorithms
O((V + E) log V) where V is the number of vertices and E is the number of edges. Each node is inserted into the heap once (O(log V)) and each edge causes a potential decrease-key operation (implemented as a new insertion, also O(log V)). With a Fibonacci heap, this can be improved to O(E + V log V), but our implementation uses Java's PriorityQueue (binary heap).
+How does the Heap Sort algorithm work step by step? Algorithms
1. Build a max-heap from the input array by heapifying all non-leaf nodes from n/2-1 down to 0. 2. Repeatedly extract the maximum element: swap the root (largest) with the last element of the unsorted portion, then heapify the reduced heap. 3. After n-1 extractions, the array is sorted in ascending order. 4. We reverse the result to get descending order for rankings.
+What is the space complexity of Heap Sort? Algorithms
O(1) auxiliary space. Heap Sort sorts in-place using the original array—no additional data structures beyond a few variables (loop counters, swap temporary). This is one of Heap Sort's main advantages over Merge Sort (O(n) space). Note: the output list in our implementation uses O(n) space for the result, but the core sorting is in-place.
+Explain the 0-1 Knapsack DP formulation. Algorithms
dp[i][w] = maximum value achievable using first i items with budget w. Recurrence: dp[i][w] = max(dp[i-1][w], dp[i-1][w-cost_i] + value_i) if item i fits (cost_i <= w), else dp[i-1][w]. Base case: dp[0][w] = 0 for all w. The answer is dp[n][W]. Backtracking from dp[n][W] identifies which items were selected.
+Can the 0-1 Knapsack be solved with recursion? Algorithms
Yes, with a recursive function: knap(i, w) = max(knap(i-1, w), knap(i-1, w-cost_i) + value_i). However, without memoization, this has O(2^n) time due to overlapping subproblems. The DP table approach uses O(n*W) time by computing each subproblem exactly once. For our 12 attractions, 2^12 = 4096 is tractable, but the DP approach demonstrates optimal substructure.
+What would happen if we used Fractional Knapsack instead? Algorithms
Fractional Knapsack (greedy by value/weight ratio) would allow visiting "parts" of attractions, which is unrealistic. It would also produce a different (higher value) result since items can be split. Fractional Knapsack has O(n log n) time (greedy by ratio) but doesn't apply to real trip planning where attractions are indivisible.

Data Structures

+What data structure does Dijkstra's algorithm primarily use? Data Structures
A min-priority queue (binary heap) to efficiently retrieve the unvisited node with the smallest distance. Java's PriorityQueue provides O(log n) insertion and O(1) peek operations. The graph is stored as an adjacency list (Map<String, List<Edge>>) for O(1) neighbor lookup.
+What is the difference between a max-heap and a min-heap? Data Structures
In a max-heap, the parent node is always greater than or equal to its children, so the maximum element is at the root. In a min-heap, the parent is always smaller than or equal to its children. Dijkstra uses a min-heap (priority queue) to extract the closest node. Heap Sort uses a max-heap to repeatedly extract the maximum element.
+Why is the DP table 2D for 0-1 Knapsack? Data Structures
The DP table has dimensions (n+1) x (W+1) where n is the number of items and W is the budget. The first dimension tracks which items are considered, the second tracks available budget. Each cell dp[i][w] stores the maximum value achievable with i items and w budget. This 2D structure captures the two decision variables: item index and remaining capacity.

Implementation

+How does the desktop Dijkstra service handle stale priority queue entries? Implementation
Java's PriorityQueue doesn't support decrease-key. When a shorter distance to a node is found, we simply add the node again to the queue. When polling, we check a visited set and skip any node already finalized. This is the standard workaround for Dijkstra with Java's PriorityQueue.
+How are ties handled in the Heap Sort ranking? Implementation
Ties are handled by the compare() method: first compare by composite score in descending order, then by name alphabetically. This ensures deterministic ordering even when two attractions have identical scores.
+How does the map route rendering work in the desktop app? Implementation
MapPanel.paintComponent() draws 12 layers in sequence: background, grid, water, districts, roads, junctions, route, markers, labels. The route is drawn as 3 overlapping strokes: a semi-transparent glow (14px), the main route line (4px green), and a white center line (1.5px). Animated white dots move along the path using time-based interpolation.
+How does the web frontend communicate with the backend? Implementation
The frontend uses the fetch() API to make HTTP requests to the Java backend on port 8080. The API client wraps these calls with JSON serialization/deserialization. The backend parses JSON using basic string extraction (extractJsonString, extractJsonInt) and responds with model-generated JSON via toJson() methods.
+What is the difference between the desktop and web algorithm services? Implementation
Desktop services cache the graph in the constructor (loaded once from DB) and use private fields with getters. Web services rebuild the graph on each request and use public fields for models. The core algorithm logic is identical. The web version of Knapsack also tracks skipped items (not done in desktop version).

Architecture

+Why does the project have both a desktop and web version? Architecture
The dual-mode approach demonstrates two different software architectures: monolithic desktop (Swing + direct service calls) and client-server (HTTP server + SPA frontend). This showcases the versatility of Java and the reusability of algorithm code. It also covers more evaluation criteria for the DAA course.
+How is the database path different between desktop and web? Architecture
The desktop app uses "jdbc:sqlite:trip.db" (relative to project root). The web app uses "jdbc:sqlite:webapp/trip.db" (relative to the webapp directory). This separation prevents data conflicts when running both versions simultaneously. The schema and seed data are identical.
+What design pattern does the web backend use? Architecture
The web backend follows a simple Controller-Service-Model pattern. TripPlannerApp.java acts as the controller (receives HTTP requests, routes to handlers). Services (DijkstraService, KnapsackService, HeapSortService) contain the business logic. Models (Attraction, Edge, PathResult, KnapsackResult) represent data structures with JSON serialization.
+How does the SPA router work? Architecture
The Router object listens for hashchange events. When the URL hash changes (#dashboard, #trip-planner, etc.), it updates the sidebar active state, shows the corresponding content page, and triggers page-specific data loading via onPageLoad(). This enables client-side navigation without page reloads.

Database

+Why is there no foreign key relationship between attractions and edges? Database
The edges table references location names as strings (fromNode, toNode), not foreign keys to attractions. This is because edges connect both attraction districts AND intermediate junctions (which are not attractions). A junction like "Main Junction" or "North Gate" doesn't exist in the attractions table. Using string references maintains flexibility.
+How are the road edges stored? Database
Each undirected road is stored as two directed edges (A→B and B→A). Each row has fromNode, toNode, and weight. This makes graph traversal simpler since we only look at outgoing edges. The graph has 20 nodes and 35 undirected pairs (70 directed rows).

Optimization & Edge Cases

+What happens if no path exists between two locations? Optimization
The PathResult has a found=false flag. The UI displays "No route available" and clears the map route. The algorithm detects this when the end node's distance remains Integer.MAX_VALUE after the priority queue is exhausted.
+What if the budget is too small for any attraction? Optimization
The Knapsack algorithm returns an empty selected list and total cost/value of 0. The UI shows an empty state message: "No attractions fit within this budget." Backtracking finds no items where dp[i][w] differs from dp[i-1][w].
+How would you scale this project for a real city? Optimization
For a real city with thousands of locations: (1) Use a spatial database (PostGIS) for efficient geo-queries. (2) Replace the adjacency list with a more memory-efficient graph representation. (3) Add hierarchical routing (A* with landmarks, contraction hierarchies) for faster pathfinding. (4) Use real OpenStreetMap data with coordinate transformations. (5) Add lazy loading for map tiles instead of rendering everything at once.
+Could the DP table in Knapsack be optimized for space? Optimization
Yes, the 2D DP table (n+1 x W+1) can be reduced to a 1D array (W+1) by iterating budget in reverse order. The recurrence dp[w] = max(dp[w], dp[w-cost] + value) works because reading dp[w-cost] from the previous iteration (lower w values) is preserved when iterating backward. This reduces space from O(n*W) to O(W).
+Is Dijkstra suitable for negative edge weights? Optimization
No. Dijkstra fails with negative edge weights because a node's distance might be reduced after it's been finalized (popped from the priority queue). For graphs with negative weights, use Bellman-Ford (O(VE)) or SPFA. Our road network only has positive weights, so Dijkstra is correct and optimal.
+How many comparisons does Heap Sort make in the worst case? Optimization
Heap Sort makes approximately 2n log n comparisons in the worst case. Building the heap takes O(n) comparisons (about 2n). Each of the n-1 extractions takes O(log n) comparisons (about 2 log n each). Total: roughly 2n log n + 2n, which is O(n log n).
